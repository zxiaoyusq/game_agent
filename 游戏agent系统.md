游戏全链路 AI Agent 产研系统
1. 项目一句话
面向游戏策划、美术、研发、运营全流程，搭建一套基于项目知识库和多模块 AI Agent 的智能产研系统，让各岗位可以在对应 Agent 中直接产出可用成果物，并通过上下游知识和产出物串联。大幅提升游戏项目从创意、验证到落地的效率。

---
2. 需求与业务本质
游戏研发的核心不是单点完成某个文档、素材或代码，而是持续把创意转化为可验证、可上线、可运营的内容。
当前游戏项目中，策划、美术、研发、运营各环节都依赖大量人工经验和跨部门沟通。一个新玩法或新活动从想法提出，到策划案、素材需求、研发实现、测试验证、运营上线，往往需要多轮沟通和反复修改。
本项目希望通过 AI Agent 和项目知识库，把项目规则、历史经验、代码结构、美术规范、运营方法沉淀下来，让 AI 不只是回答问题，而是直接参与产研交付，产出可用的初版成果物。

---
3. 当前流程与瓶颈
当前流程
策划提出玩法或活动想法
 → 编写策划案和需求文档
 → 美术拆解素材需求并制作原画、模型、视频等资源
 → 研发理解需求并开发功能模块
 → 测试和策划反复验收
 → 运营准备活动方案、公告、投放内容
 → 上线后复盘数据和用户反馈
主要瓶颈
第一，项目知识分散。策划文档、美术规范、代码说明、历史版本经验分布在不同文档、表格和工程中，查找和理解成本高。
第二，跨部门沟通成本高。策划、美术、研发、运营对同一需求的理解容易不一致，导致返工。
第三，初版产出慢。策划案、美术需求、基础代码、Demo、运营方案都需要人工从零开始整理，影响创意验证速度。
第四，研发实现成本高。很多功能模块存在重复开发和样板代码工作，研发人员大量时间消耗在基础编码和需求翻译上。
第五，美术生产链路长。原画、建模、视频等素材从需求拆解到初版产出周期较长，难以快速支撑玩法验证和运营活动。

4. AI 创新切入点
本项是围绕真实游戏产研流程，构建多个专业 Agent 模块。
每个 Agent 都对应一个真实业务环节，同时每个 Agent之间也是有机连接的，能够读取对应知识库，产出该环节可用的成果物，并把产出结果沉淀为后续Agent模块的输入。
策划侧
策划人员可以在策划 Agent 中完成玩法构思、活动规则、数值思路、任务系统、版本需求文档等工作。
策划 Agent 不只是给建议，而是直接产出可评审、可修改、可流转的策划案和需求文档。策划确认后，这些内容会成为美术 Agent、研发 Agent、运营 Agent 的重要输入。
美术侧
美术人员可以在美术 Agent 中基于策划 Agent 的产出，继续完成美术需求拆解、风格参考整理、素材清单生成、Prompt 生成和美术素材初版生产。
美术素材生成覆盖：
原画生成
角色、道具、场景概念图生成
宣传视频、玩法演示视频、活动视频生成
素材规格检查和风格一致性校验
美术 Agent 的产出可以继续提供给研发 Agent，用于 Demo 搭建、资源接入和界面表现验证。
研发侧
研发人员可以在研发 Agent 中基于项目代码、技术架构、策划文档、美术资源规范等知识，直接生成对应功能模块代码。
研发 Agent 重点能力包括：
理解项目已有代码结构
读取策划 Agent 输出的玩法规则和需求文档
读取美术 Agent 输出的素材规格和资源说明
生成客户端逻辑、服务端接口、配置结构和测试样例
生成新玩法 Demo 或新游戏 Demo
对生成代码进行解释、修改建议和问题排查
研发 Agent 的目标不是替代研发决策，而是承担大量基础编码、模块搭建和 Demo 验证工作，让研发人员把精力放在架构、关键逻辑和质量把控上。
运营侧
运营人员可以在运营 Agent 中基于策划方案、版本内容、活动目标和用户数据，生成运营活动方案、公告文案、推送内容、用户反馈分析和版本复盘报告。
运营 Agent 可以读取前面环节沉淀的策划内容、玩法卖点、美术素材和版本功能说明，保证运营表达与实际版本内容一致。

5. AI 方案架构
本项目采用“项目知识库 + 多Agent协同 + Agent 工具调用 + Workflow”的架构，因为这个 Agent系统并不像claude code或者codex这些工具 需要面对很广泛的场景，每个模块需要完成的任务会比较确定 所以目前阶段并太需要Harness架构。
该系统的核心不是让 AI 直接从一句话生成最终结果 因为生产级游戏的产出涉及到多个模块的综合，如果使用端到端的生成，很有可能在中间某个模块就已经不是最终想要的了。所以，我们采用分模块的 Agent设计，把游戏产研流程拆成多个可控模块，让不同岗位在对应 Agent 中完成自己的工作，并通过统一知识库和标准化产出物实现协作。
5.1 项目知识库
系统沉淀项目核心资料，包括：
策划文档、玩法规则、数值规范
美术风格规范、素材标准、历史资源
客户端和服务端代码结构
接口文档、配置表、工程规范
运营活动方案、版本公告、复盘报告
用户反馈、数据指标、竞品分析资料
知识库分为通用项目知识库和模块知识库。
通用项目知识库负责保存项目基础规则、版本信息、核心玩法、通用规范。
策划知识库负责保存玩法方案、活动规则、数值经验和历史需求。
美术知识库负责保存风格规范、素材标准、原画参考、建模规范和视频规范。
研发知识库负责保存代码结构、接口规范、配置格式、工程说明和常见问题。
运营知识库负责保存活动模板、公告模板、历史复盘和用户反馈。
不同 Agent 既可以读取自己的专业知识库，也可以读取上游 Agent 已确认的产出物。

5.3 典型工作流
本系统的典型工作流不是“一句话输入，AI 自动生成全部结果”，而是各岗位在自己的 Agent 模块中逐步完成工作，上游产出经过确认后，再进入下游环节。
策划模块产出
策划人员进入策划 Agent，输入玩法目标、版本方向、用户反馈或竞品参考。
策划 Agent 结合项目玩法规则、历史策划文档和数据经验，生成玩法方案、活动规则、数值思路、需求文档和验收标准。
策划人员对结果进行审核、修改和确认。确认后的策划产出会沉淀到项目知识库，成为后续美术、研发、运营工作的输入。
阶段二：美术模块承接
美术人员进入美术 Agent，选择已经确认的策划方案作为输入。
美术 Agent 根据策划内容和项目美术规范，生成美术需求拆解、素材清单、原画参考、建模需求、视频脚本和素材生成方案。
美术人员对风格、规格和可用性进行确认。确认后的美术产出会进入知识库，并提供给研发和运营使用。
阶段三：研发模块实现
研发人员进入研发 Agent，选择已经确认的策划文档、美术资源说明和对应项目代码库。
研发 Agent 结合项目代码结构、接口规范、配置规范和上下游产出，生成相关功能模块代码、配置文件、测试样例和接入说明。
对于新玩法或新方向，研发 Agent 还可以生成可运行的新玩法 Demo 或新游戏 Demo，用于快速验证创意是否成立。
研发人员对代码进行审核、运行、修改和合并，确保生成内容符合工程质量要求。
阶段四：运营模块包装
运营人员进入运营 Agent，读取已经确认的策划方案、玩法卖点、美术素材和版本功能说明。
运营 Agent 生成活动方案、公告文案、推送内容、用户沟通话术和版本复盘模板。
运营人员审核后，用于活动上线、渠道分发和后续数据复盘。
阶段五：审核与沉淀
审核 Agent 对各环节产出进行一致性检查，包括：
策划目标和研发实现是否一致
美术素材是否符合玩法和项目风格
运营表达是否与真实功能一致
代码实现是否遗漏需求点
测试验收项是否完整
最终，每个环节的确认版本都会沉淀回项目知识库，形成可复用的项目资产，让后续版本和新项目可以继续复用。

6. 项目规划
阶段一：梳理真实产研流程
梳理策划、美术、研发、运营在游戏项目中的高频工作场景，明确每个环节最适合 AI 直接产出的成果物。
重点不是一开始做大而全的平台，而是先找到可以快速验证价值的链路。
阶段二：建设项目知识库
整理已有策划文档、美术规范、代码结构、接口文档、配置规则、运营方案和历史复盘。
同时定义各模块 Agent 的输入输出格式，保证策划产出能被美术和研发使用，美术产出能被研发和运营使用，研发产出能被运营和测试理解。
阶段三：建设核心 Agent 模块
优先建设四个核心模块：
策划 Agent
美术 Agent
研发 Agent
运营 Agent
每个 Agent 先覆盖 1 到 2 个高频场景，保证能够产出可审核、可修改、可复用的初版成果物。
阶段四：打通模块串联链路
选择一个具体玩法或活动场景，打通从策划产出、美术承接、研发实现、运营包装到审核检查的完整链路。
重点验证各模块之间的产出是否能顺利流转，而不是只验证单点生成效果。
阶段五：Demo 验证与效果评估
通过人工流程和 AI Agent 流程对比，评估：
策划案产出时间
美术需求拆解时间
素材初版生成效率
研发代码生成比例
Demo 搭建周期
跨部门沟通次数
返工次数
最终产出可用率
阶段六：迭代扩展
在核心链路验证成功后，逐步扩展更多玩法类型、更多美术素材类型、更多代码模块和更多运营场景，最终形成可在多个游戏项目复用的全链路 Agent 系统。
